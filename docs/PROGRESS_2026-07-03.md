### Finding 1 — CDM root cause = missing CJK fonts (RESOLVED)
Initial full eval gave CDM 0.870 / Overall 89.06 (3 points below PARITY). Per-subset analysis showed the drag was concentrated in subsets whose formulas contain **Chinese text in `\text{}`** (e.g. `$$\frac{\text{阿里巴巴服务新消费交易额}}{...}$$`). The OmniDocBench CDM scorer selects `formular_template_zh` (which needs `CJK.sty` + the `gkai` font) by `contains_chinese(latex)` — **not** by page language. Without `texlive-lang-chinese`, those formulas' Chinese text doesn't render → CDM collapses (table_hard 0.489 with text EditDist 0.003). **Fix: `apt install texlive-lang-chinese`** → CDM 0.870→0.957, Overall 89.06→91.95. Documented in the runbook memory.

### Finding 2 — issue #55 ngram=5 fix is CATASTROPHIC on the full eval (REVERTED)
The issue #55 comment (AIwork4me) recommends `ngram_size=5, window=256, repetition_penalty=1.05` to fix text-repetition looping. Implemented + validated on the 3 looping pages (2 FIXED, 1 BOUNDED). But the **full eval crashed to Overall 64.56** (every metric ~3× worse): `ngram_size=5` globally bans legitimate 5-grams (`<|det|>` tags, bboxes, table headers, common phrases) → degrades ~1645 normal pages. The comment validated on 2 pages only; the full 1651 reveals the harm. **Reverted to `ngram=35` (the 91.95 baseline).** The ~3 looping pages (~1% drag) need a **targeted** (per-page runaway detection + truncation) approach, not a global ngram change. `src/rocm_ocr/repetition_fix.py` kept with a WARNING + the runaway length-cap idea for a future targeted fix.

## Backend B (SGLang) & C (vLLM) — ⚠️ BLOCKED on a version wall
- **vLLM**: the only prebuilt ROCm wheel is `vllm 0.24.0+rocm723`, which pins a custom `torch 2.11.0+gitd0c8b1f` (ROCm 7.2.3 build) — **newer than this host's 7.2.1 driver** (userspace cannot exceed the driver), so it won't init the GPU. The rocm723 index also lacks common deps (numpy/transformers/...), so combining it with PyPI makes uv backtrack indefinitely. No older ROCm vLLM wheel exists.
- **SGLang**: source build path (clone v0.5.9 → compile `sgl-kernel` for gfx1100 → `pip install python[all_hip]`). `sgl-kernel` compiled successfully for ROCm ✓, and v0.5.9 **does** register the DeepSeek-OCR/Unlimited-OCR model (`python/sglang/srt/models/deepseek_ocr.py`) — so arch support is there. But `[all_hip]`'s pinned deps (e.g. `torchao==0.9.0`) need a newer torch than 2.5.1 → uv backtracks.
- **Root cause**: this host's ROCm 7.2.1 driver + torch 2.5.1+rocm6.2 (the issue #55 baseline) is older than the current vLLM/SGLang ROCm stack (torch 2.11 / rocm7.x).

### Options to unblock the backends
1. **Upgrade the ROCm driver to 7.2.3** (sudo) → rocm723 torch works → vLLM installs + SGLang's deps resolve. Cleanest; unblocks both.
2. Salvage **SGLang** with a newer-but-≤7.2.1-driver torch (e.g. rocm6.3/7.0 2.7/2.8) + resolve torchao. vLLM stays blocked (needs torch 2.11).
3. Accept **PyTorch-only** (91.95, already aligned with PARITY) + document vLLM/SGLang as requiring a newer ROCm driver.
